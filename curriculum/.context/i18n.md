# Internationalization (i18n)

This document describes how the curriculum localizes its **student-facing text**. It is the
curriculum-side companion to `app/.context/i18n.md` (which deliberately puts "exercise/curriculum
text" out of scope for the app's `next-intl` catalog — that text is owned here).

## The core principle

**The curriculum owns and resolves all of its own student-facing strings, and hands finished,
already-localized strings to whoever renders them.** The interpreter and the app are dumb
renderers:

- `ExecutionContext.logicError(message)` relays the string it is given verbatim (the interpreter's
  `error.runtime.LogicErrorInExecution` is just `"{{message}}"`). The English originates in
  exercise-authored code, so the interpreter cannot own its translation.
- The app renders whatever `errorHtml` string a scenario produces. It does not parse or re-key it.

This keeps the curriculum's runtime the single place that resolves curriculum text, and keeps the
curriculum ↔ interpreter and curriculum ↔ app contracts as "a string", unchanged by localization.

## Which mechanism: i18next (not the app's ICU)

The monorepo has two i18n stacks. The curriculum joins the **interpreter's**, not the app's:

- **App (chrome):** `next-intl` + ICU messages.
- **Interpreter (language/runtime errors):** **i18next** — `interpreters/src/javascript/translator.ts`
  (`createInstance`, `translate(key, options)`, `{{var}}` interpolation, per-locale
  `locales/{locale}/translation.json`, `en` default + `fallbackLng: "en"`, `escapeValue: false`).
  JavaScript already ships a partial `hu`.

Curriculum strings are execution-layer peers of the interpreter's error strings — both are resolved
and handed to renderers — and the curriculum already depends on `@jiki/interpreters`. So the
curriculum reuses i18next as its resolution mechanism (plurals, `{{var}}` interpolation,
`escapeValue: false`). i18next supports plurals (`key_one`/`key_other`), so nothing is lost versus
ICU. (The two stacks never meet at a boundary: the curriculum hands the app a finished string, so
there is no ICU-vs-i18next dialect conflict.)

The one deliberate divergence from the interpreter: the interpreter bundles its single shared catalog
into the compiled JS, but the curriculum must **not** bundle the corpus of per-exercise dictionaries.
The deployment target is a Cloudflare Worker + R2, so loading every exercise's catalogs into the
bundle is unacceptable. Instead the curriculum **injects the dict** (see below).

## The curriculum translator

`src/i18n/translator.ts` exposes a **factory** that builds a translator bound to ONE injected
single-locale message dict. There is **no** global instance, no mutable active locale, and no
namespace registry:

```ts
export type Messages = Record<string, unknown>; // nested i18next-style tree for ONE locale
export type Translator = (key: string, params?: Record<string, unknown>) => string;
export function createTranslator(messages?: Messages): Translator;
```

`createTranslator` spins up a fresh, isolated `createInstance()` initialized with the injected dict
under a single fixed language (`resources: { x: { translation: messages } }`, `lng: "x"`,
`interpolation: { escapeValue: false }`). Translators built from different dicts never interfere. If
`messages` is empty/undefined the translator is still constructable and gracefully returns the key.

The dict is **injected into the exercise instance**, not imported at module scope. The exercise
resolves its runtime templates against that one dict and never knows the locale code — it is handed
the already-resolved-for-that-locale dict:

```ts
// src/Exercise.ts (base class)
public setMessages(messages: Messages): void {  // app injects the R2-fetched dict after construction
  this.translator = createTranslator(messages);
}
protected t(key: string, params?: Record<string, unknown>): string {
  return this.translator(key, params);        // safe default translator until injected
}
```

```ts
// inside an exercise's available function:
executionCtx.logicError(this.t("errors.unknownDice", { sides: sidesNum }));
```

The exercise's `Exercise.ts` does **not** import its `locales/*/translation.json` — those files are
authoring/build inputs (see below), never bundled.

### Locale threading (entirely app-side)

The exercise never sees a locale code. Locale selection lives **entirely in the app**: the app knows
the student's locale, fetches that exercise's per-locale message dict from R2 (as a field on the
exercise content blob, alongside instructions/stub/solution), and injects it via `setMessages`.
Switching locale means fetching a different dict and constructing a fresh exercise — there is no
`changeLanguage` seam in the curriculum.

> **The app-side wire does not exist yet.** As of this writing the app injects no dict and
> `useExerciseLoader` hardcodes `"en"` for exercise content, and the interpreter always runs in its
> default `en` (so even its own `hu` error messages are currently unreachable). Delivering the
> per-locale dict from R2 and injecting it (plus setting the interpreter's locale) is net-new
> app-side work. It is the prerequisite follow-up for any of this to show through to students.

## Catalog files

Per-exercise, co-located next to `instructions/` (which the translator does **not** touch — that is
markdown handled by the app's build pipeline):

```
src/exercises/<slug>/locales/
  en/translation.json   # canonical, required
  hu/translation.json   # stub (copy of en) until translated
```

- These are **authoring / build inputs**, read from curriculum **source** by the app's cache
  generator (the same way it reads `instructions/`, `stub.*`, `solution.*`). They are **not** imported
  into any bundled code and **not** copied into `dist` — the `files: ["dist", "src"]` in `package.json`
  already ships `src`, so the generator can read them from source.
- **`en` is canonical and required**, and there is **no runtime fallback**: the app injects exactly the
  one locale it fetched, and a missing key surfaces as the key rather than silent English. Completeness
  across locales is the guard test's job (structure parity), not a fallback's.
- **Never auto-translate.** New locales are stubbed by copying `en` verbatim; Jeremy triggers real
  translation. (Same rule as `instructions/*.md` and concept `*.md`.)
- `escapeValue: false` means `<code>…</code>` and other HTML in messages and interpolated values pass
  through untouched — safe because these strings are rendered as HTML or relayed verbatim.

### Emitting the dict into the per-locale content blob

At build time the app's cache generator emits each exercise's catalog into that exercise's per-locale
content blob (alongside `instructions`/`stub`/`solution`) and uploads it to R2. At runtime the app
fetches the blob for the student's locale and injects `blob.messages` via `setMessages`. This is why
the Worker never bundles the corpus of dictionaries: each request pulls exactly one exercise's one
locale from R2.

### Guard test

`tests/i18n/catalogs.test.ts` mirrors the app's `messages.test.ts`: it scans every exercise `locales/`
dir and asserts each has an `en` catalog, every other locale is structurally identical to `en` (same
key tree), and no message has leading/trailing whitespace. It is generic, so it covers new catalogs
automatically as the extraction proceeds.

## Scope: what is (and isn't) covered

Four kinds of student-facing string live in `BaseExerciseCore` (see `src/exercises/types.ts`) with no
locale mechanism today. All should move to this pattern:

| Source                                    | Kind             | Resolves at                                              |
| ----------------------------------------- | ---------------- | -------------------------------------------------------- |
| `errorHtml` in `scenarios.ts`             | runtime feedback | failure time (curriculum resolves, app renders)          |
| `logicError(...)` in `Exercise.ts`        | runtime feedback | execution time (curriculum resolves, interpreter relays) |
| `hints` in `metadata.json`                | static           | definition-assembly time                                 |
| scenario `name`/`description`             | static           | definition-assembly time                                 |
| `FunctionInfo` `description` (prose only) | static           | definition-assembly time                                 |

Not in scope:

- **Markdown content** — `instructions/{locale}.md` (+ its `title`/`description` frontmatter) and
  concept `*.md` stay on the app's build-time markdown pipeline. The dividing line: prose authored in
  a file → app build pipeline; strings authored as data in code → curriculum translator.
- **`FunctionInfo` `examples`/`signature`** — these are JavaScript code, not prose; they stay literal
  (JS-first rule). Only `description`/`category` prose gets keys.
- **`llm-metadata.ts`** — fed to the LLM tutor, not rendered to students. Out of scope for now.
- **`FunctionExecutionError`** (interpreter) — wraps arbitrary native JS `Error.message`; inherently
  untranslatable. Known limitation.

See also the interpreter-side plan for `interpreters/src/javascript` error/describer extraction
(`~/.claude/plans/fizzy-munching-peacock.md`); the `LogicError` pre-translated contract there is the
mirror of this document.

## CI validation (cross-package key completeness)

We want CI to prove we have valid, complete keys for **every** i18n string across app, curriculum,
and interpreters. Because only the app knows the authoritative locale set, this is validated the same
way PR #885 already validates content: **one app-side, `REQUIRED_LOCALES`-driven validator** that
`fs`-walks the curriculum (`src/exercises/*/locales/*`) and interpreter (`src/*/locales/*`) catalog
files and reuses the existing "required-locale-present + structure-parity" checks
(`app/lib/content/validator.ts`), plus a format-validity check.

Per catalog it asserts:

- every `REQUIRED_LOCALES` locale file is present (the `validateRequiredLocales` analog),
- non-`en` locales are **key-tree-identical** to `en` (the `summary`-block-parity analog),
- each message is valid in its format — **ICU** for the app, **i18next `{{…}}`** for curriculum +
  interpreters — with matching placeholder sets, and no leading/trailing whitespace,
- the interpreter `system` bundle is excluded from human-locale parity (it is a pseudo-locale).

`REQUIRED_LOCALES` (must-be-fully-stubbed) is deliberately separate from `ALL_LOCALES` (known/typed):
a locale graduates from the second set to the first when its extraction is done. That membership _is_
the enforcement tier — no per-locale flags needed.

Two levels of completeness, because curriculum/interpreters are mid-extraction (content is already
fully stubbed, so it gates strictly today):

- **Per-catalog gate (enforce now):** of the catalogs that exist, each is complete + parity-valid.
- **Corpus gate (enforce at the end):** every exercise has a catalog and every interpreter language
  has every `REQUIRED_LOCALES` locale. Turned on only after a `seed-language.js`-style pass stubs the
  missing locales (copy `en`) everywhere — the same extract → seed stubs → gate playbook content used.
  Note `seed-language.js` currently covers concepts/blog/articles but **not** exercises or these
  catalogs; extending it is part of this work.

### Merge gate

**This i18n effort is not considered done — and is not merged — until the cross-package CI validation
above is fully in place and green across all three packages** (app, curriculum, interpreters),
including the corpus gate after the extraction + `hu`-stub seeding pass. Partial extraction may land
incrementally behind the per-catalog gate, but "done" means the whole surface is covered and CI
enforces it end to end.

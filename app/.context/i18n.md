# Internationalization (i18n)

This file covers two layers:

1. **UI-string i18n** (application chrome — buttons, labels, modals, toasts) — powered by
   `next-intl`. See [UI String Translation](#ui-string-translation) below.
2. **Content localization** (blog posts, articles) — per-locale markdown + generated indexes.
   Covered from [URL Structure](#url-structure) onward.

All app-chrome UI strings have been extracted into the message catalog. The original
area-by-area sweep is documented in `.context/i18n-ui-strings-plan.md` (the runbook).

## UI String Translation

UI strings use **next-intl** (v4) in "without i18n routing" mode: next-intl itself doesn't do URL
routing, but `resolveLocale` derives the locale per the precedence below so public pages are
URL-driven and app pages are user-driven.

- **Library wiring**: `next.config.ts` applies `createNextIntlPlugin("./lib/i18n/request.ts")`.
- **Request config**: `lib/i18n/request.ts` loads `messages/{locale}.json` for the resolved locale.
- **Message catalog**: `messages/en.json` (canonical) and `messages/hu.json`. `hu` falls back to
  English per the `?? en` convention. The two files are structurally identical (same key tree).
- **Key typing**: `global.d.ts` augments next-intl's `AppConfig.Messages` so message keys
  autocomplete and typecheck against `messages/en.json`.

### Locale resolution model

The guiding split is **cacheable/public pages resolve locale from the URL; auth-gated per-user
pages resolve from the user's preference** (`user.locale` via `/internal/me`). The `NEXT_LOCALE`
cookie is an optimistic, FE-owned SSR hint that self-heals from `/me` — it is never the source of
truth.

- **Server resolution** (`lib/i18n/resolveLocale.ts`), precedence:
  1. Explicit URL locale segment — surfaced by `middleware.ts` as the `x-url-locale` request header
     (`URL_LOCALE_HEADER`). Middleware always sets or clears it from the trusted path, so it can't be
     spoofed. Wins for everyone, keeping public/cacheable pages deterministic per URL.
  2. Otherwise, if the auth-presence cookie exists (possibly-logged-in), the `NEXT_LOCALE` cookie
     (the optimistic seed).
  3. Otherwise (anon, naked URL) `DEFAULT_LOCALE`. The `NEXT_LOCALE` cookie is intentionally ignored
     here so anon cacheable pages stay deterministic.
- **Provider + client correction** (`components/i18n/ClientLocaleProvider.tsx`): `app/layout.tsx`
  resolves `getLocale()`/`getMessages()` server-side and passes them as `initialLocale`/
  `initialMessages`. The provider renders that seed for first paint (so external/anon SSR is
  unaffected), then watches `user.locale` from the auth store. When the authoritative locale differs
  from the seed it dynamic-imports that catalog and swaps in place. "Swap pending"
  (`authoritative !== active`) is derived synchronously and the provider renders `LoadingJiki` during
  a swap, so the app never paints in the wrong locale; a failed catalog load is recorded so the swap
  settles instead of spinning forever. For anon users `user` is null, so no swap ever occurs.
- **Cookie writes** (`lib/i18n/localeCookie.ts` `setLocaleCookie`): `authStore.setUser` mirrors the
  authoritative `user.locale` into `NEXT_LOCALE` (covers `/me`, login, signup, confirm). The cookie
  can be stale but never lastingly wrong, since `/me` re-seeds it.
- **`<html lang>`**: set server-side in `app/layout.tsx` from the seed; `ClientLocaleProvider`
  corrects `document.documentElement.lang` after a swap.
- **Signup seeding** (`lib/i18n/detectSeedLocale.ts`): `SignupForm` sends `locale` inside the `user`
  registration params **only when the user is on a locale page** (a non-default `useLocale()`);
  otherwise it omits the field and the backend falls back to the request's `Accept-Language` header
  (the same data `navigator.languages` would give, read once server-side) then the default. The
  backend persists a supported value and ignores unsupported ones.

### Backend contract

- `/internal/me` returns `user.locale` (required, never null for a real row — the backend
  lazy-backfills from `Accept-Language` on first read). `User.locale` is typed required.
- `POST /auth/signup` accepts an optional `user.locale` and ignores unsupported values.
- `PATCH /internal/settings/locale` persists an explicit change (no in-app switcher UI yet — see
  deferred).

### Using translations

- Server components: `const t = await getTranslations("namespace")` from `next-intl/server`.
- Client components: `const t = useTranslations("namespace")` from `next-intl`.
- Rich text: `t.rich("key", { strong: (c) => <strong>{c}</strong> })`.
- Interpolation/plurals: ICU syntax in the JSON (`{name}`, `{count, plural, ...}`).

Reference implementation: `components/layout/ExternalFooter.tsx`.

Key-naming convention: namespace = feature folder; dot-nested to mirror the component tree;
camelCase leaf keys describing the role (not the English text). Shared strings live under `common.*`.

### Catalog namespaces

Top-level namespaces in `messages/en.json` (and `hu.json`, which is structurally identical):
`common`, `layout`, `auth`, `settings`, `toasts`, `premium`, `roadmap`, `landing`, `seo`, `modals`,
`misc`. `seo.*` backs page `generateMetadata` (titles/descriptions); `toasts.*` covers the
component/hook-reachable toast messages.

### Not yet localized (deferred)

These render English regardless of locale and are intentionally out of the chrome sweep:

- **Plain-`.ts` strings with no hook/await context**: `lib/pricing.ts` (`PRICING_TIERS` names,
  descriptions, feature lists — consumed across many components), `lib/subscriptions/handlers.ts`
  and `components/settings/subscription/handlers.ts` (toast strings in plain async functions),
  `lib/settings/settingsStore.ts` (Zustand store toasts),
  `components/coding-exercise/lib/orchestrator/TestSuiteManager.ts` (class). Localizing these needs
  the pass-in pattern (translate at the call site and pass the string down) or a follow-up.
- **`SubscriptionErrorBoundary`** — a class component; `useTranslations` can't run in its render.
- **`app/global-error.tsx`** — replaces the entire HTML tree (no `NextIntlClientProvider` in scope),
  so it keeps hardcoded English.
- **Content** (out of scope by design): blog/article/concept bodies, exercise/curriculum text,
  `components/testimonials/testimonials.json`, and the blog/articles `[locale]` routing.
- **In-app language switcher**: no settings UI to change locale yet (and no live catalog swap on a
  settings change). `settingsStore.updateLocale` exists but is unused. Deferred.
- **Locale-specific auth pages**: `/auth/*` are naked (not locale-prefixed), so `useLocale()` there
  is the default for anon users and signup seeding relies on the backend `Accept-Language` fallback.
  When locale-prefixed auth pages are added, signup seeding picks them up with no code change.
- **Locale-aware chrome links**: header/nav links stay naked. Only `blog`/`articles` have `[locale]`
  routes (the content grids already build `/hu/...` links from their `locale` prop); other chrome
  targets have no locale route, so prefixing them is deferred until those routes exist.

## Locale Configuration

Supported locales are defined in `lib/locales.ts`:

```typescript
export const SUPPORTED_LOCALES = ["en", "hu"] as const;
export type Locale = (typeof SUPPORTED_LOCALES)[number];
export const DEFAULT_LOCALE: Locale = "en";
```

## URL Structure

- **Default locale (English)**: Served at naked URLs (`/blog`, `/articles`)
- **Non-default locales**: Served at locale-prefixed URLs (`/hu/blog`, `/hu/articles`)
- **English locale prefix**: Redirects to naked URLs (`/en/blog` → `/blog`)

## Adding a New Locale

When adding a new locale, you must:

1. Add the locale code to `SUPPORTED_LOCALES` in `lib/locales.ts`
2. Add translated content markdown files in the content package
3. **Generate a search index for the new locale** - The content generation script automatically creates search indexes per locale at `public/static/search/articles-{locale}.json`. Run `pnpm run content:generate` after adding content.

## Search Indexes

Article search indexes are generated at build time, one per locale:

- `public/static/search/articles-en.json`
- `public/static/search/articles-hu.json`

These are automatically created by `scripts/generate-content.js` when content is generated. When adding a new locale with articles, the corresponding search index will be created automatically.
